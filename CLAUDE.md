This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

`run` is a CLI runtime built with Cobra: it turns commands defined in YAML files (`.run.yaml`, or the directory form `.run/run.yaml`) into a command-line interface and executes them, resolving the local command file from the current directory (with ancestor traversal) and merging it with the global `~/.config/run/run.yaml` (local top-level names shadow global ones).

## Build Commands

```sh
make build   # Build binary to ./bin/run (with version ldflags)
make test    # Run tests
make fmt     # Format code
make vet     # Vet code
make tidy    # Tidy dependencies
make clean   # Remove build artifacts
```

Binary name is read from `.product_name`.

## Release

```sh
make release type=patch|minor|major            # dry run (default)
make release type=patch dryrun=false           # create and push tag
make re-release [tag=vX.Y.Z] dryrun=false      # re-release an existing tag
```

Pushing a `v*` tag triggers `.github/workflows/gorelease.yml`, which builds multi-platform binaries with GoReleaser and uploads them to GitHub Releases.

## Architecture

- `main.go` — entry point, calls `cmd.Execute()`
- `cmd/` — Cobra root command. Built-in features live under the single reserved `self` subcommand (`run self list`, `run self version`, `run self completion <shell>`, `run self path [target]`), so every other bare argument is a command name
  - `root.go` — `run <command> [subcommand... [args...]]` resolves the argument path through nested commands via rootCmd's `RunE`; no args shows the command list. Config loading is layered (`loadConfigFiles`): every file located by `config.Find` is loaded and validated (a broken global file is a hard error everywhere) and kept as a `configFile` (`cfg` + `workDir`) in precedence order. The first path segment picks the origin file — local shadows global at top-level-name granularity, so a whole subtree always comes from one file — and that file supplies the top-level `env:`/`shell:` and the working directory for everything below it; listing and completion use `mergedCommands` (top-level overlay, higher-precedence files win). Path resolution is greedy: names matching a subcommand are path segments, the rest become arguments; a literal `--` (split manually in `runCommand`, since `SetInterspersed(false)` keeps it in `args`) forces the boundary — tokens after it are always literal, never options. Processing order: split at `--` → path resolution → `--help` interception → `applyOptions` on the pre-`--` remainder → `applyArguments` on positionals + literals → normalized option tokens appended last. A `--help` anywhere in the pre-`--` remainder renders the command's declared help (`commandHelp`) instead of running it — checked before `resolveEnv` so help never evaluates dynamic values; a declared option named `help` opts the command out, and `run cmd -- --help` forwards a literal `--help` (`-h` is never special: single-dash tokens stay positional). `applyOptions` extracts declared long-form options (`--name`, `--name=value`, `--name value`; only for commands with an `options:` declaration — others pass through untouched, unknown `--x` is an error otherwise), builds per-option env vars (bool `true`/`false`, value options fall back to default or empty; every declared option always gets an entry), and re-emits recognized options in declaration order after all positionals so `$1..$n` stay stable and `"$@"` forwards everything (defaults materialize; unset bools and valueless options are omitted). `applyArguments` validates CLI arguments against a command's `arguments:` declaration, fills defaults, and builds per-argument environment variables. Environment variables from `env:` declarations are accumulated unevaluated (`config.Value`) during path resolution (top-level `Config.Env`, then each matched command's `Command.Env`, inner overrides outer, declared arguments/options highest — they cannot collide, validation forbids shared names); the shell resolves along the same walk (top-level `Config.Shell`, each matched command's non-empty `Command.Shell` overrides, empty means `sh`) and is passed to `runner.Run` and every dynamic-value `runner.Capture` call; `resolveEnv` materializes them only when the command executes: literals pass through, dynamic values run via `runner.Capture` in name order seeing the OS env plus literal entries only (no dynamic-to-dynamic references; overridden dynamic entries never run). Argument/option defaults (`*config.Value`) resolve lazily through a closure passed to `applyOptions`/`applyArguments` — only when the default is actually used — with the fully resolved env visible, so defaults can reference shared env values. The merged result is emitted as a sorted deterministic `name=value` list by `envList`; plain string values are literal — run never expands `$VAR` in them, `{run: ...}` is the explicit opt-in for computed values. Shell completion (`completeCommands`, the root `ValidArgsFunction` — cobra's `__complete` passes every typed token through, including `--`-prefixed ones, because `SetInterspersed(false)` disables cobra's own flag completion once an arg exists) mirrors the greedy path resolution: on the path it completes subcommand names, past it a `-`-prefixed word completes the resolved command's declared options plus `--help` (declaration order, `\t`-descriptions, already-typed options skipped via `completeOptions`); value positions (`awaitingOptionValue` for the space form, `=` in the word), positional positions, and anything after a literal `--` get no candidates, and nothing ever falls back to file completion. Cobra's default `help`/`completion` subcommands are disabled; since cobra insists on a help command once subcommands exist (and always offers it in shell completion), `selfCmd` doubles as the help command so no extra name is reserved or completed (side effect: `self` is absent from `--help`'s Available Commands, which is why the Long text names it). Exit codes are propagated via `runner.ExitError` in `Execute()`
  - `self.go` — the reserved `self` namespace and its subcommands (`list`, `version`, `completion`, `path`); a new built-in feature means a new subcommand here, never a new reserved top-level name. `run self path [root|local|global]` (default `root`) prints a directory for use in run strings (e.g. `. "$(run self path local)/lib.sh"`): `root` = the local file's workDir, `local` = the directory containing the local command file (`.run/` for the directory form), `global` = `config.GlobalDir()` (printed without an existence check). `root`/`local` pick the first non-`Global` `config.Find` result and error when there is none; a `$RUN_CONFIG` file counts as local, and `global` is an error while `$RUN_CONFIG` is set
  - `list.go` — command listing helpers (`runList`, `listCommands`); nested commands are flattened with space-joined paths, followed by the declared argument signature (`<required>` / `[defaulted]`) and option signature (`[--bool]` / `[--name <name>]`)
  - `help.go` — per-command help rendering (`commandHelp`): description, usage line(s) reusing `argumentSignature`/`optionSignature`, Arguments/Options sections (label column manually padded — rows without a detail stay unpadded, so no tabwriter), Commands section reusing `listCommands`. Defaults render via `defaultLabel`: literals inline, `{run: ...}` as `(default: dynamic)` — help never executes anything. An implicit `--help` row is appended unless the command declares a `help` option (`declaresOption`)
- `internal/config/` — YAML schema (`Config`, `Command`; commands nest via `Command.Commands`), loading/validation (`config.go`: `loadFile` decodes with `KnownFields(true)`, so an unknown key anywhere is a load error — yaml.v3 accumulates all of them in one `TypeError` with line numbers; an empty file decodes as an empty config (`io.EOF` swallowed) and fails later in `Validate` with "no commands defined"), and command file resolution (`finder.go`: `Find` returns `File`s — path + workDir + a `Global` flag marking the merged global file — in precedence order: `$RUN_CONFIG` alone if set; otherwise the ancestor-search local file followed by the global `~/.config/run/run.yaml` (workDir = cwd, `Global` = true), both when both exist. The local file has two interchangeable forms checked per ancestor level — `.run.yaml`/`.run.yml` and `.run/run.yaml`/`.run/run.yml` — with workDir = the ancestor directory in both cases (never `.run/` itself); both forms in the same directory is a hard error, `.yaml` silently wins over `.yml` within a form. `GlobalDir` returns `~/.config/run` without an existence check, for `Find` and `run self path global`). Files referenced via `includes:` (allowed at the top level and on any command) are eagerly merged at load time (`expandIncludes`): the included file's commands land flat in the including scope, name collisions are an error, relative paths resolve against the including file's directory, cycles are detected via an absolute-path chain, and `Includes` is cleared after expansion so the rest of the code only ever sees an inline command tree. An included file's top-level `env:` is pushed down into each command it defines (the command's own `env:` wins), so it never leaks to sibling commands in the including file; its top-level `shell:` is pushed down the same way. `env:` keys are validated (non-empty, no `=`) via `validateEnv`. `env:` values and argument/option defaults are `config.Value`: a plain scalar is literal, the `{run: ...}` mapping form is dynamic (a shell command whose stdout, trailing newlines trimmed, becomes the value at execution time); the shape (`run` as the only key, non-empty) is enforced in `Value.UnmarshalYAML`. `options:` declarations (`Option`: name/description/type/default; `type: bool` vs value option) are validated via `validateOptions`: require `run:`, names non-empty/unique/no `=`/no leading `-`, no collision with `arguments:` names (both become env vars), `type` limited to empty/`string`/`bool`, and bool options may not declare a default (no `--no-name` form exists, so unset must mean false). `Validate` rejects a top-level command named `self` (the reserved built-in namespace); nested commands may use the name
- `internal/runner/` — executes `run:` strings with `<shell> -c` (the shell parameter; empty means `sh`); arguments become the shell's positional parameters (`$1`, `"$@"`; `$0` is `run`) and extra env entries are appended after `os.Environ()` (os/exec keeps the last duplicate, so they override the inherited environment). `ExitError` carries the command's exit code. `Capture` runs a string the same way but returns its stdout with trailing newlines trimmed (used for dynamic values); its failures are plain errors, never `ExitError`, so a captured command's exit code can't masquerade as the run command's own
- `internal/version/` — version info injected via ldflags at build time

Key behavior:

- When a local `.run.yaml` (or `.run/run.yaml`) is found by ancestor search, its commands run in the directory containing the file — for the `.run/` form, the directory containing `.run/`, so `includes:` resolve against `.run/` while execution stays at the project root (like make/just); merged global commands run in the current directory. `$RUN_CONFIG` disables merging — that file is used alone
- The local and global files are merged with local top-level names shadowing global ones; each file's top-level `env:`/`shell:` applies only to its own commands
- There is deliberately no built-in sourcing/prelude feature: run strings that need shared shell functions source them explicitly (`. "$(run self path local)/lib.sh"`), keeping both the run string and the library file complete, shellcheck-able shell
- Commands may define `run`, nested `commands`, or both (validated recursively after include expansion). A command with both runs its own `run` string when invoked directly; a group without one lists its subcommands
- Flags must come before the command name (`SetInterspersed(false)`); everything after the first non-flag argument is part of the command path
